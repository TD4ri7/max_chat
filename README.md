# MAX Chat

MAX Chat — это веб-приложение для общения, которое позволяет пользователям взаимодействовать друг с другом в режиме реального времени[cite: 3]. Приложение поддерживает глобальный чат, приватные диалоги и групповые чаты[cite: 3]. Проект разработан с учетом принципов распределенных приложений[cite: 3].

## 1. Описание проекта

MAX Chat — это современное интерактивное веб-приложение для общения, разработанное для обеспечения взаимодействия пользователей в режиме реального времени. В сегодняшнем цифровом мире быстрая и надежная коммуникация имеет критическое значение, поэтому приложение спроектировано так, чтобы предлагать как глобальное общение, так и более приватные решения. Пользователи могут участвовать в открытом глобальном чате, создавать безопасные приватные диалоги между двумя людьми и инициировать групповые чаты для больших компаний.

Более широкая цель проекта заключалась в применении на практике принципов создания распределенных приложений. Это включает в себя оптимизацию обмена данными между клиентом и сервером, управление базами данных и контроль асинхронных процессов. Приложение ориентировано на простой и интуитивно понятный пользовательский интерфейс, обеспечивая при этом стабильность и безопасность системы на заднем плане.

### 1.1 Функциональные требования
Функциональные требования описывают специфическое поведение и функции системы, которые программное обеспечение должно выполнять:
- **Управление пользователями:** Система должна позволять регистрацию новых пользователей, обеспечивая использование уникальных имен пользователей и адресов электронной почты. Система аутентификации должна обеспечивать безопасный вход и завершение сессии (выход из системы).
- **Управление профилем:** У каждого зарегистрированного пользователя должна быть возможность изменять свои личные данные (например, описание «О себе») и загружать персональную фотографию профиля (аватар), которая сохраняется на сервере.
- **Каналы общения:** Приложение должно поддерживать глобальный чат, где все активные пользователи могут обмениваться сообщениями. Кроме того, должна быть возможность создавать приватные (1-на-1) беседы и групповые чаты с несколькими участниками.
- **Передача сообщений:** Система должна обеспечивать передачу текстовых сообщений и мультимедиа (изображений) в реальном времени во всех типах чатов.
- **Модерация контента:** Пользователи с правами администратора должны иметь возможность безвозвратно удалять из системы неподобающий контент или сообщения, нарушающие правила.

### 1.2 Нефункциональные требования
Нефункциональные требования определяют свойства и стандарты качества системы:
- **Надежность и доступность:** Приложение должно быть способно выдерживать несколько активных пользователей одновременно без заметного снижения производительности.
- **Безопасность:** Пароли пользователей должны быть зашифрованы в базе данных (например, с помощью bcrypt). Обмен данными между клиентом и сервером должен быть защищен, а управление сессиями должно предотвращать несанкционированный доступ.
- **Удобство использования:** Пользовательский интерфейс должен быть интуитивно понятным, отзывчивым (responsive) и безупречно работать в самых распространенных веб-браузерах (Chrome, Firefox, Safari, Edge) как на компьютерах, так и на мобильных устройствах.
- **Масштабируемость:** Архитектура должна быть построена таким образом, чтобы в будущем можно было легко добавлять новые функции (например, видеозвонки, обмен файлами) и увеличивать ресурсы сервера.

### 1.3 Требования пользователей
Ожидания и потребности пользователей в отношении системы следующие:
- Пользователь ожидает, что процесс регистраций и входа будет быстрым и понятным, без лишних сложностей.
- Пользователь хочет, чтобы отправленные сообщения и изображения доходили до адресата без задержек.
- Пользователь хочет легко находить других зарегистрированных в системе пользователей и начинать с ними диалоги.
- Пользователь предполагает, что его приватные чаты недоступны для других, а данные защищены.

## 2. Описание системы

MAX Chat построен на модели клиент-сервер (client-server), где бизнес-логика и обработка данных строго отделены от пользовательского интерфейса. Это обеспечивает лучшую производительность, поддерживаемость и безопасность системы. Система является модульной и состоит из независимых сервисов, которые взаимодействуют друг с другом через определенные интерфейсы.

### 2.1 Используемые технологии
Для реализации проекта был выбран современный и широко поддерживаемый технологический стек, оптимизированный для создания веб-приложений реального времени:
- **Node.js:** Асинхронная среда выполнения JavaScript на стороне сервера, основанная на событиях, которая идеально подходит для ресурсоемких приложений ввода-вывода (таких как приложения для чата).
- **Express.js:** Минималистичный и гибкий фреймворк веб-приложений для Node.js, который упрощает создание конечных точек REST API и управление HTTP-запросами.
- **Fetch API:** Клиентский интерфейс для выполнения HTTP-запросов к серверу для асинхронной загрузки данных без перезагрузки страницы.
- **FormData:** Стандартная веб-технология, позволяющая конструировать пары ключ/значение для передачи файлов (например, аватаров профиля и сообщений с изображениями) на сервер.

### 2.2 Языки программирования
Основными языками программирования проекта являются:
- **JavaScript (JS):** Используется как на стороне сервера (Node.js), так и на стороне клиента. Обеспечивает единство системы и позволяет повторно использовать код.
- **HTML5:** Обеспечивает структуру и семантику веб-страницы, поддерживая современные мультимедийные элементы.
- **CSS3:** Отвечает за визуальное представление, макет и адаптивный дизайн (responsive design) приложения.

### 2.3 Базы данных и платформы
- **База данных:** Система использует MongoDB. MongoDB — это гибкая NoSQL документная база данных, которая хранит данные в JSON-подобных (BSON) документах. Это особенно подходит для приложений чата, поскольку структуры данных (например, сообщения, участники чата) могут быть динамическими. Динамическая схема позволяет осуществлять быструю разработку.
- **Платформа:** Система спроектирована как веб-приложение на базе браузера (Web App). Она не требует от пользователя загрузки или установки специального программного обеспечения в операционную систему, что делает приложение независимым от платформы (работает на Windows, macOS, Linux).

### 2.4 Общая структура системы
Система разделена на три основных логических слоя:
1. **Слой представления (Клиентское приложение):** Код, выполняемый в браузере пользователя, который управляет интерфейсом, реагирует на действия пользователя (нажатия кнопок, заполнение форм) и рендерит данные, полученные от сервера.
2. **Слой бизнес-логики (Сервер):** Сервер Node.js, который принимает запросы клиентов, проверяет права, обрабатывает данные (например, сохранение изображений в файловую систему), применяет правила приложения и формирует ответы.
3. **Слой данных (База данных):** MongoDB, которая отвечает за безопасное хранение и быстрый доступ к постоянной информации (пользователи, хэш-значения паролей, история чатов).

### 2.5 Процессы и API
Обмен данными между клиентом и сервером происходит через API, следующий архитектурному стилю REST (Representational State Transfer). Запросы и ответы стандартизированы в формате JSON. Ниже приведены наиболее важные конечные точки REST API системы.

| Метод | URL | Описание и данные |
| :--- | :--- | :--- |
| **POST** | `/api/auth/register` | Регистрация нового пользователя. Требует email, имя пользователя и пароль. В базу данных сохраняется зашифрованный пароль. |
| **POST** | `/api/auth/login` | Вход пользователя. При успешной аутентификации создается серверная сессия. |
| **POST** | `/api/auth/logout` | Выход из системы. Удаление активной сессии пользователя. |
| **GET** | `/api/auth/me` | Возвращает данные текущего вошедшего пользователя на основе сессии. |
| **PUT** | `/api/profile` | Обновление текстовых данных профиля пользователя. |
| **POST** | `/api/profile/avatar` | Загрузка аватара пользователя. Требует объект FormData с файлом. |
| **GET** | `/api/chats/dialogs` | Возвращает список всех чатов, в которых участвует активный пользователь. |
| **POST** | `/api/chats/group` | Создание нового группового чата с указанием названия и участников. |
| **POST** | `/api/messages` | Отправка сообщения (текст или изображение) в конкретный чат. |
| **GET** | `/api/messages/:id` | Загружает хронологическую историю сообщений чата с указанным ID. |
| **DELETE** | `/api/messages/:id` | Удаление сообщения (требуются права администратора или статус владельца сообщения). |

### 2.6 Модели данных и UML-диаграммы
Структура базы данных реализована с помощью трех основных коллекций в MongoDB. Ниже описана структура моделей.

#### Модель: User (Пользователь)
- `_id`: (ObjectId) Уникальный идентификатор.
- `username`: (String) Уникальный никнейм, выбранный пользователем.
- `email`: (String) Unikaalne e-posti aadress, mida kasutatakse ka sisselogimisel.
- `password`: (String) Безопасно сохраненный пароль в виде хэша.
- `about`: (String) Краткое описание пользователя.
- `avatar`: (String) URL или путь к файлу аватара профиля.
- `role`: (String) Роль пользователя в системе (например, `"user"`, `"admin"`).

#### Модель: Chat (Чат)
- `_id`: (ObjectId) Уникальный идентификатор чата.
- `type`: (String) Тип чата (`"global"`, `"private"`, `"group"`).
- `participants`: (Array of ObjectIds) Ссылки на модель User, участвующих в чате.
- `title`: (String) Название группового чата (не требуется для приватных чатов).
- `createdAt`: (Date) Временная метка создания чата.

#### Модель: Message (Сообщение)
- `_id`: (ObjectId) Уникальный идентификатор сообщения.
- `sender`: (ObjectId) Ссылка на модель User, отправившего сообщение.
- `chatId`: (ObjectId) Ссылка на модель Chat, к которой относится сообщение.
- `text`: (String) Текст сообщения.
- `imageUrl`: (String) Путь к прикрепленному изображению (если есть).
- `createdAt`: (Date) Точная временная метка отправки сообщения для обеспечения хронологии.

> *Примечание: Рекомендуется добавить UML-диаграмму классов и диаграмму прецедентов использования в репозиторий.*

## 3. Дизайн и архитектура

Архитектура программного обеспечения определяет расположение частей системы, связи между ними и способы взаимодействия. Правильный архитектурный выбор обеспечивает читаемость кода и более легкое управление в долгосрочной перспективе.

### 3.1 Архитектура программного обеспечения
MAX Chat основан на многослойной модульной архитектуре, аналогичной шаблону MVC (Model-View-Controller):
- **View (Представление):** HTML, CSS и клиентский JavaScript. Отображает данные и перехватывает действия пользователя.
- **Controller (Контроллеры):** Роутеры Express.js и функции логики, которые обрабатывают HTTP-запросы и координируют работу моделей.
- **Model (Модели данных):** Схемы Mongoose, которые напрямую взаимодействуют с базой данных MongoDB и заботятся о структуре и валидации данных.

### 3.2 Описание модулей
Кодовая база сервера разделена на специфические модули во избежание монолитного и сложного в обслуживании кода:
- **Модуль аутентификации (Auth Module):** Этот модуль содержит всю логику для регистрации пользователей, криптографии (сохранения паролей), входа в систему и создания/управления сессиями.
- **Модуль профиля (Profile Module):** Обрабатывает загрузку файлов. Включает проверку типа и размера файлов для предотвращения попадания вредоносных файлов на сервер. Этот модуль взаимодействует с файловой системой для сохранения изображений и с базой данных для обновления URL-адресов.
- **Модуль чатов (Chat Module):** Управляет бизнес-логикой при создании новых чатов. Проверяет количество участников, настраивает права и предотвращает создание дублирующих приватных чатов.
- **Модуль сообщений (Message Module):** Модуль с самым интенсивным трафиком. Отвечает за прием сообщений, запись в базу данных, валидацию и последующее чтение, всегда предоставляя клиенту данные в правильном хронологическом порядке.

### 3.3 Интерфейсы и обмен данными
Все взаимодействия между клиентом и сервером происходят безопасно по протоколу HTTP. Стандартом обмена данными выбран JSON (JavaScript Object Notation), легкий и простой в парсинге формат данных. Для передачи бинарных данных (например, аватаров) используется кодировка `multipart/form-data` с помощью объекта FormData.

Чтобы гарантировать безошибочный обмен данными, сервер использует стандартные коды состояния HTTP (200 OK при успешном запросе, 201 Created при создании нового ресурса, 400 Bad Request при неверных данных, 401 Unauthorized при отсутствии сессии и 500 Internal Server Error при непредвиденных ошибках сервера).

## 4. Тестирование

Тестирование программного обеспечения является неотъемлемой частью системной разработки, целью которой является поиск и устранение ошибок до того, как они дойдут до конечных пользователей. Для обеспечения качества системы быловедено тщательное функциональное тестирование.

### 4.1 План тестирования
Цель плана тестирования — проверить, работают ли все функциональные требования, определенные в главе 1.1, должным образом, и может ли система справляться с некорректными (invalid) входными данными. Тестирование проводилось вручную, имитируя действия конечного пользователя в различных сценариях.

В следующей таблице приведены наиболее важные тест-кейсы (Test Cases) и их шаги.

## Модели данных

### User (Пользователь)
- `_id`: ObjectId (Уникальный идентификатор)[cite: 3]
- `username`: String (Уникальное имя пользователя)[cite: 3]
- `email`: String (Уникальный email)[cite: 3]
- `password`: String (Хешированный пароль)[cite: 3]
- `about`: String (Краткое описание "о себе")[cite: 3]
- `avatar`: String (Путь к файлу изображения)[cite: 3]
- `role`: String (Роль в системе: `"user"` или `"admin"`)[cite: 3]

### Chat (Чат)
- `_id`: ObjectId[cite: 3]
- `type`: String (Тип чата: `"global"`, `"private"` или `"group"`)[cite: 3]
- `participants`: Array (Ссылки на модель User — участники)[cite: 3]
- `title`: String (Название группового чата)[cite: 3]
- `createdAt`: Date (Время создания)[cite: 3]

### Message (Сообщение)
- `_id`: ObjectId[cite: 3]
- `sender`: ObjectId (Ссылка на модель User — отправитель)[cite: 3]
- `chatId`: ObjectId (Ссылка на модель Chat)[cite: 3]
- `text`: String (Текст сообщения)[cite: 3]
- `imageUrl`: String (Путь к изображению, если оно прикреплено)[cite: 3]
- `createdAt`: Date (Временная метка отправки)[cite: 3]
